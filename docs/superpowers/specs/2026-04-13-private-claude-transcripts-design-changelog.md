## Architect Review — 2026-04-13 (Pass 2)

Pass 2 asked: which of these earn their keep?

### What landed

The seven changes in row 2 of the convergence table above.

The single most important finding pass 2 surfaced was **not** an over-engineering removal — it was a missed concern: pass 1's `https://x-access-token:TOKEN@github.com/...` clone URL puts the PAT in `.git/config` _and_ in the `git clone` argv where a same-user process can read it from `/proc/<pid>/cmdline` for the duration of the clone. The fix is to use `git -c http.extraHeader="Authorization: Basic ..."` instead, which keeps the token off the persistent remote URL and out of `.git/config` entirely. This is the standard pattern `actions/checkout`'s own auth helper uses.

- **`RUN_ATTEMPT` in filename + env** — re-runs of a single failed run are a real GHA workflow concept and would otherwise produce misleading filenames. Cheap and correct.
- **Splitting verification gate 2 into failure-path and success-path dry runs** — the split is meaningful: it forces the success path to be exercised at least once before the upstream merge. Pass 2 considered collapsing them but the gap they close is real.
- **Rollback section** — short, accurate, and surfaces the "PAT/private repo don't need to be touched on rollback" point that an implementer would otherwise have to re-derive.
- **Three new risks added by pass 1** (unbounded growth, dependency on personal repo, concurrent push race) — all kept; the race risk was rewritten to align with the no-retry decision but the underlying surface still belongs in the register.

## Architect Review — 2026-04-13 (Pass 3)

### Pass 3 focus

Targeted re-validation of the credential-handling code that pass 2 introduced, plus an edge-case sweep of the bash control flow, the YAML composition, and any orphaned references left behind by two passes of rewrites. Pass 2 was mostly reverting pass 1 over-engineering plus catching the `/proc/cmdline` leak — pass 3 asked: did pass 2 catch _all_ of the credential-leak surface, and is the spec internally consistent now that two large rewrites have settled?

### What landed

One substantive change, captured in row 3 of the convergence table above. The change is small but materially closes a real leak path:

- pass 2 correctly switched from URL-embedded PAT to `git -c http.extraHeader="Authorization: Basic <base64>"`, which keeps the raw token out of `.git/config` and out of the persistent clone URL.
- pass 2 _did not_ notice that the base64 form of `x-access-token:<TOKEN>` is **not** automatically masked by GitHub Actions. GHA's secret masker is content-based — it looks for the literal raw bytes of `secrets.X` in log lines and redacts them to `***`. A base64 transformation produces _different_ bytes that the masker doesn't recognize. So if the workflow ever ran with `set -x`, or if a future git release printed `-c` config values in error messages, the `Authorization: Basic eC1hY2Nlc3MtdG9rZW46...` string would render verbatim into a public log line and could be base64-decoded back to the raw PAT in one step.
- The fix is two lines: assign the base64 to a `B64` variable, then `echo "::add-mask::$B64"` _before_ the variable is used in any other command. GHA processes the `::add-mask::` workflow command and from that point forward redacts any occurrence of `$B64` in log output. The `::add-mask::` directive itself is consumed by the workflow runner and not echoed verbatim.
- While in the area, pass 3 also tightened the comment to call out that `printf` is a bash builtin (no fork → no subprocess argv with the raw token) and that `base64` reads from stdin (not argv), so the encoding pipeline itself is also argv-clean.

### What pass 3 reviewed and intentionally did NOT change

- **`extraHeader` syntax.** Verified `git -c http.extraHeader="Authorization: Basic <base64>"` is the correct invocation. `actions/checkout`'s own `git-auth-helper.ts` uses the identical form. The `Basic` scheme over `x-access-token:<PAT>` is GitHub's documented HTTP Basic Auth shape.
- **`git -c` repetition on both clone and push.** Confirmed `git -c` is per-invocation only — it does NOT persist into the cloned repo's `.git/config`. The spec correctly repeats `-c http.extraHeader=...` on both the `clone` and the `push`. The intermediate `git add` and `git commit` need no auth (no network), and the committer-identity `-c` flags on the commit are also correctly per-invocation.
- **Order of the `no_transcript` and `no_token` checks.** Pass 3 considered swapping them so that gate 2's failure-path verification is robust against Claude crashing before writing the transcript. Decided to leave as-is: when there is literally nothing to archive, reporting `no_transcript` is more accurate than `no_token` (the missing token is irrelevant when there's nothing to push), and gate 2's expected scenario is "Claude runs successfully, transcript exists, secret unset" — the order is correct for the _expected_ path of every gate.
- **Shallow clone + push correctness.** `git clone --depth 1` followed by adding new commits on top of HEAD and pushing is the well-trodden happy path. The historical "shallow push" git issues are about pushing branches whose ancestry crosses the shallow boundary, which is not what this script does. `--filter=blob:none` would be a marginal speedup but adds complexity and isn't needed at 52 commits/year.
- **`mkdir -p "$WORK/transcripts"` defensive line.** Pass 3 considered adding a `[ -e ... ] && [ ! -d ... ] && rm` guard against the unrealistic case where someone hand-commits a _file_ named `transcripts` into the private repo. Decided not worth a line — if Igor does that, the loud `mkdir -p` failure is the right signal, not silent recovery.
- **`github.run_attempt` availability in `env:` bindings.** Confirmed via GHA expression docs: the `github` context, including `run_attempt`, is available everywhere `${{ }}` expressions are evaluated, including step-level `env:`.
- **`gh` CLI auth state in the new step.** The new step doesn't call `gh` at all — it uses `git` with an explicit `LOGS_TOKEN` env binding scoped to this step. There is no shadowing of the other steps' `GITHUB_TOKEN`-based auth.
- **Orphaned references from pass 1/pass 2 reverts.** Grepped for `cancelled`, `retry`, `trap`, `gate 5`, `attempt loop`, `3-attempt`. Every match is intentional explanatory text (Section 2 "Single push, no retry" comment block, Section 3 rationale for the missing `cancelled` branch, Non-goals retry bullet, Risks concurrent-push entry). Nothing dangling.
- **YAML composition.** Mentally pasted the Section 2 snippet at the insertion point in `.github/workflows/changelog.yml` (between `Generate Changelog` and `Summarize Claude run` — the `Upload Claude transcript` step is removed). The snippet uses `- name:` at column 0 in the spec, which the implementer indents by 6 spaces to fit the `steps:` block. Step keys (`name`, `if`, `id`, `continue-on-error`, `env`, `run`) are valid and ordered conventionally.

### Assessment

Ready for implementation. Pass 3 made one real fix (mask the base64 credential) and otherwise found a settled spec. The convergence trajectory is 8 → 7 → 1 substantive changes, which is the diminishing-churn pattern pass 3 was looking for. The credential-handling section now accounts for all three leak surfaces (config persistence, argv persistence, log-line transformation), and pass 2's `extraHeader` decision plus pass 3's `add-mask` reinforcement together match what `actions/checkout`'s auth helper does.
